# JIRA Ticket Creation

## Behavior

Before creating any ticket:
1. Analyze the referenced code or description thoroughly
2. Ensure connectivity with JIRA MCP
2. Identify and surface any requirement gaps (ambiguous scope, missing edge cases, unclear ownership)
3. Present your gap analysis and the proposed ticket draft
4. Always present the synopsis of the changes at each step and ask for confirmation before proceeding to the next step
5. Do not include TypeScript code samples in the ticket body.
6. Ask for explicit confirmation before creating or updating the ticket on JIRA

---

## Gap Analysis (pre-creation)

Before drafting, check for and flag:
- [ ] Unclear or missing acceptance criteria
- [ ] Ambiguous scope boundaries (what's in vs. out)
- [ ] Unidentified dependencies or affected systems
- [ ] Missing non-functional requirements (performance, security, accessibility)
- [ ] No clear owner or team assignment

Present gaps as questions, e.g.:
> "Before I create this ticket, I need to clarify: How does the behavior differ for each user role? 
> What are the impacted components within the codebase?
> Are there any performance requirements we should consider?
> Any other queries "

[What exists today — reference specific files, components, or behaviors]

[What needs to change and why — business or technical motivation]

**Scope**  
[What is explicitly in scope. Call out what is NOT in scope if ambiguous.]

[Other tickets, services, or teams that must be involved or notified]
